# Traceability Matrix — Reference File

## 1. Purpose

The Requirements Traceability Matrix (RTM) is the backbone connecting every document in the V-model package. It provides objective evidence that every user requirement has been decomposed into specifications and verified through qualification protocols — and that nothing was invented along the way without a traceable origin.

From the vendor's perspective, the RTM is how you prove coverage to your customer's QA team. During a vendor audit, the auditor will open the traceability matrix first. If they can trace any URS requirement forward through FS, DS, and into IQ/OQ/PQ with a passing status — and do so in under sixty seconds — you pass the sniff test. If they cannot, every other document comes under scrutiny.

The traceability matrix is a **computed artifact**. It is generated by `scripts/traceability_check.py` from the project manifest, never hand-authored or manually maintained. Manual matrices drift from reality within days. The manifest is the source of truth; the matrix is its rendered view.

---

## 2. Predecessor Dependencies

The RTM grows with each document added to the project. It should be regenerated after every document approval.

| Stage | What the Matrix Can Show |
|---|---|
| After URS approved | URS requirements listed, all forward traces empty — establishes the coverage baseline |
| After FS approved | URS -> FS links populated, untraced URS requirements visible as gaps |
| After DS approved | URS -> FS -> DS chain complete, depth gaps detectable |
| After IQ approved | DS -> IQ verification links, untested DS items visible |
| After OQ approved | FS -> OQ verification links, untested FS items visible |
| After PQ approved | URS -> PQ verification links, full chain completeness assessable |

The matrix is useful at every stage, not only when the full package is complete. Early generation surfaces gaps when they are cheap to fix.

---

## 3. Matrix Format

Each row represents one traceable requirement. Columns span the full V-model chain:

| URS ID | URS Text | FS ID(s) | DS ID(s) | IQ ID(s) | OQ ID(s) | PQ ID(s) | Status |
|---|---|---|---|---|---|---|---|
| URS-001 | System shall authenticate users via SSO | FS-003, FS-004 | DS-007 | IQ-012 | OQ-005 | PQ-002 | PASS |
| URS-002 | System shall generate audit trails for all data changes | FS-008 | DS-015 | IQ-020 | OQ-011 | -- | GAP |

The `--` marker indicates a missing link. The Status column reflects the worst-case roll-up across all linked verification items (see Section 8).

When a URS requirement traces to multiple FS items, the row expands. The matrix always anchors on URS requirements as the left-most column — this is the auditor's starting point.

---

## 4. Four Traceability Link Types

Traceability is bidirectional. GAMP 5 and IEEE 830 both distinguish four link directions:

### Forward-to (Decomposition)
Traces a requirement **down** into its implementing specification. URS-001 forward-traces to FS-003. This answers: "How is this requirement addressed?"

### Forward-from (Verification)
Traces a specification **down** into its verification protocol. FS-003 forward-traces to OQ-005. This answers: "How is this specification tested?"

### Backward-to (Origin)
Traces a specification **up** to its source requirement. FS-003 backward-traces to URS-001. This answers: "Why does this specification exist?"

### Backward-from (Justification)
Traces a test protocol **up** to its source specification. OQ-005 backward-traces to FS-003. This answers: "Why does this test exist?"

Every link in the manifest encodes both directions implicitly. When URS-001 lists `traces_to: [FS-003]`, the generation script infers that FS-003 has a backward-to link to URS-001. The manifest stores forward links; backward links are computed.

### Why All Four Matter

An auditor may enter the matrix from any direction:
- "Show me that every URS requirement has a test" (forward-to, then forward-from)
- "This test failed — what requirement is affected?" (backward-from, then backward-to)
- "Why does this FS item exist? What user need does it serve?" (backward-to)
- "This design component has no test — is that intentional?" (forward-from, gap)

## 5. Three Audience Views

The same traceability data serves three audiences with different needs. The generation script produces all three views from a single manifest.

### Engineer View (Detailed)
Full matrix with every ID, requirement text, link chain, gap annotations, and status. Includes TODO markers and gap remediation suggestions. Used during document generation and coached refinement.

### Auditor View (Compliance)
Clean matrix without TODO markers or coaching annotations. Shows requirement IDs, abbreviated text (first 80 characters), trace links, verification status, and gap count per severity level. Summary counts by severity appear at the top. This view ships in the published .docx as the traceability appendix.

### Management View (Dashboard)
Aggregate metrics only — coverage percentages, gap counts by severity, RED/AMBER/GREEN status, trend indicators. No individual requirements. Fits on one page. Used for status meetings, go/no-go decisions, and executive reporting.

---

## 6. Gap Analysis — Five Gap Types

The generation script classifies every traceability gap into one of five types. Each has a distinct meaning and remediation path.

### Untraced Requirement
A URS item with no forward trace to any FS item. The requirement exists, but nothing in the specification addresses it. **Remediation:** Add FS items that decompose the requirement, or document a rationale for exclusion (out-of-scope, deferred to future release).

### Untested Specification
An FS or DS item with no corresponding test protocol. The specification exists, but nothing verifies it. **Remediation:** Add test cases in the appropriate protocol (OQ for FS items, IQ for DS items), or justify omission with a risk-based rationale per GAMP 5 2nd Edition.

### Orphan Test
A test case that traces to no requirement or specification. Effort without justification — the test may be valid but it is not traceable, which means an auditor cannot assess its purpose. **Remediation:** Link the test to an existing requirement, or identify a missing requirement that the test implicitly covers.

### Broken Trace
A referenced ID that does not exist in the manifest. This is a data integrity failure — a copy-paste error, a deleted requirement whose references were not cleaned up, or an ID scheme change that left stale references. **Remediation:** Correct the ID reference or remove the broken link. Broken traces block publish.

### Depth Gap
A requirement that skips a V-model level. For example, a URS item traces directly to an OQ test case without passing through an FS item. The requirement may be tested, but there is no specification documenting how it is implemented. **Remediation:** Add the intermediate specification, or document why the skip is acceptable (rare — typically only for simple configuration-level requirements).

---

## 7. Gap Severity Classification

Every gap is assigned one of four severity levels. Severity determines whether a gap blocks publish, triggers a warning, or is noted for housekeeping.

### Critical
A safety-critical or regulatory requirement has no verification. This is an audit showstopper. Examples: 21 CFR Part 11 electronic signature requirement with no OQ test; data integrity requirement (ALCOA+) with no verification protocol. **Publish behavior:** Block.

### Major
A core functional requirement is missing test coverage. Not safety-critical, but an auditor will raise it as a finding. Examples: A key workflow requirement in the URS with no OQ coverage; an FS interface specification with no verification. **Publish behavior:** Warn, require acknowledgment.

### Minor
A non-critical requirement has incomplete traceability. The requirement is addressed somewhere, but the trace chain is not fully linked. An auditor may note it as an observation. Examples: A "nice to have" URS requirement with no forward trace; a DS item for a non-critical utility with no IQ coverage. **Publish behavior:** Warn.

### Informational
ID convention inconsistencies, stale references that do not break traces, or cosmetic issues. Not audit-relevant, but cleanup improves professionalism. Examples: Non-sequential ID numbering; a requirement with two IDs referencing the same FS item (redundant link, not broken). **Publish behavior:** Note in report only.

## 8. Status Roll-Up Logic

Each requirement row in the matrix carries a status derived from its linked verification items.

### Per-Requirement States

| State | Meaning |
|---|---|
| NOT_STARTED | Requirement exists but no linked tests have been executed |
| IN_PROGRESS | At least one linked test exists but not all are complete |
| PASS | All linked verification items have passing results |
| FAIL | At least one linked verification item has a failing result |
| GAP | One or more expected trace links are missing (any gap type from Section 6) |
| BLOCKED | A predecessor requirement's failure prevents this item's verification |

### Transition Rules

Status is computed, never manually set. The rules in priority order (highest wins):

1. If any linked test has status FAIL -> requirement status is **FAIL**
2. If any gap of severity Critical or Major exists -> requirement status is **GAP**
3. If a predecessor requirement in the trace chain has status FAIL -> **BLOCKED**
4. If all linked tests have status PASS and no gaps exist -> **PASS**
5. If at least one linked test exists but not all are complete -> **IN_PROGRESS**
6. If no linked tests exist and no gaps are flagged -> **NOT_STARTED**

FAIL takes precedence over GAP because a failing test is a known problem; a gap is a potential problem. Both demand action, but a failure has concrete evidence.

## 9. Aggregate Roll-Up

Project-level and document-level summaries aggregate individual requirement statuses into actionable indicators.

### Coverage Percentage
`(Requirements with at least one forward trace) / (Total requirements) * 100`

A coverage percentage below 100% means requirements exist with no specification addressing them. Target: 100% for vendor-owned documents.

### Verification Percentage
`(Requirements where all linked tests PASS) / (Total requirements) * 100`

Verification percentage tracks how much of the specification is proven to work. This number starts at 0% and climbs as protocols are executed.

### RED / AMBER / GREEN Thresholds

| Indicator | Coverage % | Verification % | Critical Gaps | Major Gaps |
|---|---|---|---|---|
| GREEN | 100% | >= 95% | 0 | 0 |
| AMBER | >= 90% | >= 75% | 0 | <= 3 |
| RED | < 90% | < 75% | > 0 | > 3 |

These thresholds are defaults. Override them in the manifest under `overrides.traceability.thresholds` if your customer has stricter or more lenient criteria.

## 10. Metrics

The generation script computes five metrics included in the management dashboard view.

### Coverage Rate
Percentage of requirements with complete forward trace chains through all applicable V-model levels. Differs from coverage percentage (Section 9) by requiring **full depth**, not just one link.

### Volatility Index
`(Requirements added, modified, or deleted in the last manifest update) / (Total requirements) * 100`

High volatility late in the project signals scope instability. A volatility index above 15% after FS approval warrants a scope review. Above 25% after OQ approval is a red flag — your customer's validation lead will want to understand why.

### Stability Index
`100 - Volatility Index`. The inverse view — management prefers "95% stable" over "5% volatile."

### Orphan Rate
`(Orphan tests) / (Total test cases) * 100`

Non-zero orphan rates are common during early development when tests are written before formal requirements are registered. The rate should converge to 0% before publish.

### Untested Rate
`(Requirements with no linked test in any protocol) / (Total requirements) * 100`

The untested rate should reach 0% before the validation package ships. Any non-zero value at publish time requires a documented risk-based justification per GAMP 5 2nd Edition.

## 11. Generation Rules

The traceability matrix is produced by `scripts/traceability_check.py`. It is never hand-authored.

### Input
The project manifest at `docs/v-model/v-model-manifest.json`. The script reads the `requirements` array and `documents` status map.

### Processing Steps

1. **Parse** — Load all requirements from the manifest. Build an in-memory graph of ID -> traces_to -> tested_by relationships.
2. **Resolve** — Walk every forward link. For each target ID, verify it exists in the manifest. Flag broken traces immediately.
3. **Infer backward links** — For every forward link A -> B, register backward link B -> A. The manifest stores only forward links; backward links are computed.
4. **Detect gaps** — Apply the five gap type checks (Section 6) to every requirement node.
5. **Classify severity** — For each gap, determine severity based on the requirement's regulatory tags and criticality (Section 7).
6. **Compute status** — Apply roll-up logic (Section 8) to every requirement row.
7. **Aggregate** — Compute project-level metrics (Section 9, Section 10).
8. **Render** — Output three views: engineer (detailed), auditor (compliance), management (dashboard).

### When It Runs

- After every document approval (Phase 3 of the generation cycle)
- On demand via the "Check Traceability" entry point
- Before any publish operation (mandatory safeguard)

### Output
Structured report with gaps grouped by severity, then by gap type. Includes the full matrix in the requested view and all computed metrics.

## 12. Common Problems

### Trace Rot
Links that were correct when written but have drifted as requirements evolved. A URS requirement was reworded, its FS decomposition was updated, but the DS and OQ items still reference the old interpretation. **Prevention:** Regenerate the matrix after every document approval. The script detects when requirement text has changed but downstream items have not been reviewed.

### ID Drift
Requirement IDs that were renumbered, reformatted, or had their prefix changed without updating all references. Common when teams migrate from spreadsheets to structured tools, or when the customer requests a different ID scheme mid-project. **Prevention:** The ID immutability rule (Section 13). Once assigned, IDs never change. If the scheme must change, the script provides a bulk migration command that updates all references atomically.

### Many-to-Many Explosion
A single URS requirement traces to many FS items, each of which traces to many DS items, each of which traces to many test cases. The matrix becomes unreadable, and gap analysis produces false positives because the combinatorial surface makes it hard to determine whether coverage is truly complete. **Prevention:** Decompose complex URS requirements into atomic sub-requirements before tracing. If URS-001 traces to more than five FS items, it is likely a compound requirement that should be split. The coached refinement phase flags this pattern.

---

## 13. ID Immutability Rule

Once a requirement ID is assigned and registered in the manifest, it is **permanent**. IDs are never reused, renumbered, or reformatted.

If a requirement is deleted, its ID is retired — it remains in the manifest with status `WITHDRAWN` and a rationale. The ID slot is never reassigned to a new requirement. This prevents a catastrophic class of traceability errors where a test case linked to the old URS-005 silently becomes linked to a new, unrelated URS-005.

If the ID prefix scheme changes (e.g., customer requests `PROJ-URS-001` instead of `URS-001`), use the bulk migration in `traceability_check.py` which atomically updates every reference across all documents and the manifest in a single operation. Never rename IDs by hand.

## Anti-Patterns

These patterns indicate traceability problems. The reviewer agent checks for them during coached refinement.

| Anti-Pattern | What It Looks Like | Why It Is Wrong |
|---|---|---|
| Copy-paste traces | Every URS item traces to the same FS item | Indicates the FS was not properly decomposed; traces are cosmetic |
| One-to-one mirroring | URS-001 -> FS-001 -> DS-001 -> OQ-001 with identical text | Suggests the team duplicated requirements across documents instead of decomposing them |
| Test-first tracing | Tests written first, requirements backfilled to match | Produces requirements that describe the system as-built, not as-needed; auditors detect this |
| Selective tracing | Only "easy" requirements are traced; complex or ambiguous ones are left unlinked | Creates a false sense of coverage; auditors check the hard requirements specifically |
| Phantom coverage | A test case exists and is linked, but it does not actually verify the requirement it claims to | Requires human review; the script flags test cases whose text has no semantic overlap with their linked requirement |

## Coaching Prompts

Use these during Phase 2 (coached refinement) when traceability issues are detected:

- "URS-{N} traces to five FS items. That usually means the requirement is compound — can we split it into atomic sub-requirements?"
- "FS-{N} has no backward trace. It exists in the spec but no user requirement justifies it. Is this an implied requirement we should add to the URS, or scope creep we should remove?"
- "This OQ test case references DS-{N}, which is a design-level item. OQ should verify FS-level behavior. Should this be an IQ test instead?"
- "The traceability matrix shows 100% coverage but 0% verification. Coverage without executed tests is a paper exercise — your customer's QA team will not accept it."
- "Three URS requirements all trace to FS-{N}. That means one functional specification is carrying three user needs. If FS-{N} changes, all three requirements are affected. Consider whether it should be decomposed."

## GAMP 5 Addenda

### Risk-Based Traceability Depth

GAMP 5 2nd Edition allows risk-based decisions about traceability depth. Not every requirement needs the same level of trace rigor:

| GAMP Category | Expected Trace Depth |
|---|---|
| Category 3 (Non-configured) | Lightweight — IQ verification of correct installation is often sufficient. OQ/PQ may be reduced or eliminated with documented rationale. |
| Category 4 (Configured) | Standard — Full URS -> FS -> OQ chain. DS and IQ coverage focused on configuration verification. |
| Category 5 (Custom) | Comprehensive — Full chain through all V-model levels. No shortcuts. Every custom-built component needs specification and verification. |

### CSA (Computer Software Assurance) Considerations

Under FDA's 2025 CSA guidance, the level of assurance activities is determined by risk to patient safety, not by a blanket requirement for exhaustive documentation. The traceability matrix supports this by:
- Tagging requirements with risk classification (the manifest `requirements` array supports a `risk` field)
- Allowing the management dashboard to filter metrics by risk tier
- Enabling risk-justified omissions with documented rationale rather than requiring 100% trace coverage for low-risk items
